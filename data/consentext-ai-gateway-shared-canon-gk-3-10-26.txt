Purpose

Section 1 – Core Gateway Definition
Working Definition
The Consentext AI Gateway is the governed interaction layer that determines how a user’s health context may be used with AI, routes each request through the appropriate privacy-capability lane, and preserves control through minimization, boundaries, and auditability.

Gateway Core Responsibilities
The Gateway has three core jobs:
1. Classify the request
Determine what kind of task the user is asking for.
2. Apply exposure logic
Determine what level of privacy, capability, and control is appropriate.
3. Route the request
Send the request to the correct lane.

Canonical Flow
User Request    ↓Gateway Classification    ↓Exposure Decision    ↓Lane Selection    ↓Processing    ↓Response    ↓Audit Record
Both prototypes should preserve this conceptual flow.

Lane 1 – Deterministic
Canonical Name
Deterministic
Short Definition
No AI is used. The system handles the request using structured logic, calculations, retrieval, thresholds, timelines, or templated summaries.
Purpose
This lane exists for tasks that do not require AI reasoning.
Typical Task Types

timeline generation
structured summaries
calculations
threshold alerts
missing-data surfacing
list / table generation
Example User Intent

“List my lab values by date”
“Show what changed over six months”
Data Exposure
internal only
no external AI
no external export
Prototype Provider Label
none
deterministic engine
internal structured logic
Strategic Meaning
This is the trust anchor.
Canonical User Message
“This request can be handled without AI.”

Lane 2 – Private
Canonical Name
Private
Short Definition
AI is used internally, but sensitive personal health context stays inside the protected Consentext boundary.
Purpose
This lane exists for tasks where AI explanation is helpful, but raw sensitive context should not leave the protected environment.
Typical Task Types
explanation of personal labs
explanation of personal records
plain-English interpretation of personal health context
summarization of sensitive health data
understanding personal patterns without broad external export
Example User Intent
“Explain my lab results”
“Help me understand what changed”
“Summarize this health pattern for me”
Data Exposure
internal protected context only
no external export
Prototype Provider Label
Qwen
internal protected model
protected internal AI
Strategic Meaning
This is the highest-privacy AI lane.
Canonical User Message
“This request uses protected internal AI. Sensitive data stays inside Consentext.”

Lane 3 – Private+
Canonical Name
Private+
Short Definition
External AI may be used, but only after the gateway generates a reduced, minimum-necessary context packet.
Purpose
This lane exists for questions that benefit from broader AI assistance but do not require sharing the full protected context.
Typical Task Types
lifestyle improvement questions
habit questions
general guidance questions
reduced-context support questions
non-diagnostic improvement exploration
Example User Intent
“What lifestyle changes may improve these labs?”
“What habits may affect this pattern?”
“What general actions should I explore?”
Data Exposure
reduced packet only
external export allowed in minimized form
Prototype Provider Label
external AI
reduced-context AI
minimized export AI
Strategic Meaning
This is the adoption bridge.
Canonical User Message
“This request uses a reduced context packet for external AI assistance.”

Lane 4 – Max Intelligence
Canonical Name
Max Intelligence
Short Definition
Broader governed AI interaction is allowed for requests that require stronger external AI capability.
Purpose
This lane exists for open-ended, comparative, or research-oriented questions where broader AI capability is useful.

external AI

Strategic Meaning

Canonical User Message
“This request uses the highest-capability governed AI lane.”

Canonical Name
Off-Board

Section 3 – Authoritative Routing Logic
This section defines the baseline routing patterns both prototypes should follow.
These rules do not need to be perfect.They need to be consistent.

Routing Principle 1
If the request can be answered with structured internal logic, it should route to Deterministic.
Routing Principle 2
If the request asks AI to explain sensitive personal health context, it should route to Private.
Routing Principle 3
If the request asks for general improvement ideas or behavior suggestions that can work from minimized context, it should route to Private+.
Routing Principle 4
If the request is broader, comparative, research-oriented, or open-ended, it should route to Max Intelligence.
Routing Principle 5
If the user bypasses Consentext, the state is Off-Board, not a governed lane.

Routing Tie-Break Rule
If a prompt seems ambiguous between two lanes:
prefer more private if explanation is enough
prefer Private+ over Max Intelligence if reduced context seems sufficient
prefer Deterministic whenever AI is not needed
This keeps the prototype aligned with the governance-first philosophy.

Section 9 – Closing Guidance
The purpose of this canon is not to lock the final product forever.
It is to keep the next prototype round aligned enough that the founders can learn from both builds without confusion.
The key thing to preserve is this:
The gateway is the governed layer that decides how health context may interact with AI.
Everything else in the prototype should reinforce that.
